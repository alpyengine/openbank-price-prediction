El ciclo, en 4 frases

1
Creas un batch de predicciones (Openbank AI: "este ticker valdrá X en 1M / 3M / 6M / 12M"). Se guarda en la tabla batches.
2
Cada sábado una Edge Function apunta el cierre real del viernes de cada ticker → tabla weekly_prices. Así ves la evolución semana a semana.
3
Cuando una predicción vence (llega su fecha objetivo), otra Edge Function de madrugada (Mar–Sáb) busca el precio real de ese día y decide si acertó: hit / close / miss → actualiza batches.
4
Redes de seguridad: un cron de recuperación el lunes, un watchdog Lun+Jue y emails de aviso. Y cada 2 semanas tú revisas a mano lo único que ningún cron puede detectar: que un cron no llegue a ejecutarse.

Dónde se rompía — y cómo se arregló (v7.9.0)

El problema (antes de v7.9.0): el rol de los crons tenía statement_timeout = 120 s. Twelve Data free solo permite 8 llamadas/min, por eso había un pg_sleep(8) entre cada ticker. La cuenta salía así: 120 s ÷ 8 s ≈ 15 tickers por pasada. Con carteras grandes, un cron nunca terminaba en una sola ejecución y moría por timeout justo dentro del pg_sleep. Y al ir más rápido, los 429 se registraban como "sin precio".
La solución (v7.9.0): el fetch salió de Postgres a dos Edge Functions. Un cron por minuto (dentro de su ventana) las dispara con net.http_post (asíncrono → devuelve al instante, sin tocar el timeout). Cada llamada procesa ≤7 tickers/horizontes, así la cadencia por minuto es el límite de 8/min. Detectan los 429 de verdad y son resumibles (cada pasada coge solo lo que falta, también entre días). Los crons SQL viejos (jobs 1 y 2) quedan pausados como respaldo.

Verifícalo tú mismo contra la base de datos real

Hoja ya cotejada con la BD real (jun 2026). Guarda estas consultas para volver a comprobarlo más adelante si cambias algún cron:

-- 1) Todos los crons activos y su horario
select jobid, schedule, jobname, active
from cron.job
order by jobid;

-- 2) Todas tus funciones
select proname
from pg_proc
where pronamespace = 'public'::regnamespace
order by proname;

-- 3) v7.9.0 — crons edge activos / viejos pausados
select jobid, jobname, schedule, active
from cron.job
where jobname like 'fetch-%'
order by active desc, jobid;

Si algún horario o función no coincide con lo de arriba, dímelo y actualizo el mapa.
