Diseño — Edge Function de precios

v7.9.0 · implementado, desplegado y probado · jun 2026

🎯

El problema y la decisión

Estado: implementado en v7.9.0. Las dos Edge Functions están desplegadas, probadas y con sus crons activos (jobs 10 y 12); las funciones SQL viejas quedan pausadas como respaldo (jobs 1 y 2).

Problema: el fetch de precios se ejecutaba dentro de Postgres, como una sola orden, y Postgres corta cualquier orden a los 120s. Como traer N tickers tarda minutos (límite de Twelve Data: 8 por minuto), no cabía. Además, el throttle (pg_sleep) iba demasiado rápido y provocaba errores 429 que el código etiquetaba mal como "no price obtained".

Decisión: sacar el fetch de Postgres y llevarlo a una Edge Function (gratis, sin subir de plan). El despertador (cron) solo dispara; el trabajo se hace fuera.

Clave: el throttle ya no es un sleep dentro del código — es la cadencia del cron (1 paquete por minuto (≤7), por debajo del tope de 8/min de Twelve Data). Cada ejecución es minúscula.

Programación: ventana fija de disparos por minuto (sábado 10:00–11:59 UTC = 120 disparos), no atada al nº de tickers. Cuando ya no falta nada, los disparos sobrantes son no-ops instantáneos. Si algún día se pasara de 800 US, continúa al día siguiente tras el reset de cupo (sábado + domingo + lunes ≈ 2.400/semana de margen). El número de semana se calcula por batch; el precio del viernes es el mismo y se reparte a todos los batches que contienen el ticker.

Por qué este diseño esquiva los dos muros

MuroLímiteCómo lo esquiva
Postgres (statement_timeout)120s por ordennet.http_post es asíncrono: el cron dispara y vuelve al instante. El trabajo ya no corre dentro de Postgres.
Edge Function (wall clock)400s por ejecución, y 150s para responderCada ejecución hace solo 1 paquete de ≤7 → termina en segundos, lejísimos de cualquier tope. Nunca hay sleeps largos dentro.
Twelve Data (rate limit)8 créditos/min · 800/día1 paquete de ≤7 por minuto, por debajo del tope de 8/min. Cero 429.

Reparto el trabajo en muchas piezas diminutas en vez de una pieza grande. Así ningún tope (ni el de Postgres, ni el de la Edge, ni el de la API) llega a activarse.

⚙️

Qué hace la Edge Function en cada ejecución

1
Calcula el viernes de referencia de esta semana.
2
Busca hasta 7 tickers únicos que AÚN no tengan el precio de esta semana (consulta tipo not exists sobre weekly_prices).
3
Si no falta ninguno → sale inmediatamente (no-op). Esto es lo que la hace barata aunque el cron suene cada minuto.
4
Separa US y EU. Los US en una sola llamada batch a Twelve Data (/price?symbol=A,B,C…). Los EU por Yahoo (sin límite diario).
5
Detecta errores de verdad: si la respuesta trae status:error o code (p. ej. 429), lo marca como fallo real — no como "sin precio". (Arregla el bug de los 26.)
6
Guarda los precios buenos en weekly_prices (en todas las combinaciones ticker×batch) y registra en fetch_log.
7
Sale. El minuto siguiente, el cron vuelve a disparar y coge los siguientes 7 que falten.

Es exactamente la lógica de tu usePriceFetch.js (paquetes de 8 + batch endpoint), pero el "esperar 62s" se sustituye por "ejecutarse una vez por minuto".

♾️

Reanudable y escalable (tu pregunta del "y si no caben en un día")

Como cada ejecución solo coge lo que aún falta, el sistema se reanuda solo: si una pasada no termina (por la hora, por el cupo, por lo que sea), la siguiente continúa donde se quedó. La tabla weekly_prices es el marcador de progreso. No hace falta lógica de estado extra.

🛟

Manejo de errores

429 (rate limit): ya no debería pasar (8/min respetado), pero si pasa, se detecta y se deja como "falta" → lo reintenta el minuto siguiente. Nunca se confunde con "sin datos".
Corte de red puntual (caso ALB): el ticker se queda "falta" y se reintenta solo en la siguiente pasada.
Símbolo sin datos de verdad: tras N reintentos, se marca y se avisa por email (notify_fetch_failure), con el motivo correcto.
🔁

Alcance: las dos funciones de fetch (ambas migradas)

El mismo patrón sirve para las dos, con una diferencia:

A
fetch-weekly-prices (precios del viernes) → migrada primera. Una fecha común (el viernes) para todos, así que usa el batch de Twelve Data (US) en una sola llamada. job 10
B
fetch-expired-horizons (evaluar vencidos) → migrada después. Cada horizonte tiene su propia fecha objetivo, así que no hay batch: ≤7 búsquedas individuales por minuto. El veredicto (hit/close/miss…) se calcula en SQL (save_expired_verdict), idéntico al original. job 12

Se migró A, se validó, y luego B. Las funciones SQL viejas quedan desactivadas como red de seguridad.

Decisiones tomadas

1. Proveedor para EU → Yahoo (sin límite diario). La app sigue usando Alpha Vantage para precios en vivo; las Edge Functions usan Yahoo para EU.
2. Ventanas → semanal sábado 10:00–11:59 UTC (job 10); vencidos mar–sáb 02:00–03:59 UTC (job 12). Un disparo por minuto.
3. Orden → primero fetch-weekly-prices, validada, y después fetch-expired-horizons. Ambas ya en marcha.
